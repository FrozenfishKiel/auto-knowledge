# 企业运营知识自动沉淀模块 PRD

## 1. 背景与问题

企业在使用 AI 客服、运营助手、内部问答助手时，知识库通常依赖人工维护：业务同学整理 FAQ、运营手动上传文档、管理员定期补充资料。这个流程的问题是维护成本高、更新滞后，并且大量真实业务知识实际已经出现在日常聊天记录里，却没有被沉淀下来。

在 Open WebUI 当前系统中，聊天记录、用户、用户组、知识库、向量检索、模型调用链都已经存在。我们希望基于这些现有能力，新增一个“企业运营知识自动沉淀模块”：系统按日或按周从指定范围的运营聊天记录中抽取高价值问答，经过清洗、脱敏、去重、LLM 结构化提炼和人工审核后，写入公司专属知识库。

这个功能的目标不是简单保存聊天记录，而是把真实业务对话转化为可检索、可追溯、可审核、可持续更新的企业知识资产，让系统随着运营使用逐步变得更懂公司业务。

## 2. 产品定位

本功能定位为 Open WebUI 的一等业务模块，名称暂定为 **Auto Knowledge**。

它不适合做成独立软件，因为独立软件需要重复建设用户体系、权限体系、聊天数据同步、知识库写入、向量检索和管理后台，成本高且容易割裂体验。

它也不适合只做成普通插件。Open WebUI 的插件更适合扩展模型调用、工具调用、过滤器或局部工作流；而本功能需要访问核心聊天数据表、调度任务、知识库写入链路、后台审核页面、权限配置和运行审计，属于跨后端、数据库、前端管理端的系统能力。

因此它应该作为内置模块实现：后端新增数据模型、服务层、路由和调度执行器；前端新增管理员配置页、候选知识审核页和任务运行记录页；知识库写入复用现有 Knowledge/RAG 能力。

## 3. 目标用户

主要用户是公司内部管理员、运营负责人、客服主管和知识库维护人员。

管理员负责开启模块、配置采集范围、选择目标知识库、设置执行周期和审核策略。运营负责人或客服主管负责审核 AI 提炼出的候选知识，决定是否入库。普通员工和客服不直接操作该模块，但他们的业务对话可以在符合规则和权限的前提下成为知识来源。

最终受益者是使用公司 AI 助手的员工和客户服务人员。随着真实问题不断被沉淀进知识库，后续问答命中率会提升，重复问题处理成本会下降。

## 4. 业务目标

第一个目标是减少知识库人工维护成本。管理员不再完全依赖手动导入文档，而是可以让系统周期性从真实业务对话中生成候选知识。

第二个目标是提高知识库的新鲜度。运营场景中的新政策、新活动、新异常案例、新客户问题可以在每天或每周的沉淀任务中被发现，而不是等人工整理。

第三个目标是保证知识质量和安全。聊天记录不能原样进入知识库，必须经过过滤、脱敏、去重和审核。系统要能记录候选知识来自哪些聊天和消息，方便追溯来源。

第四个目标是形成可演示的闭环：一段运营聊天发生后，定时任务抓取它，生成候选 FAQ，管理员审核通过，内容进入知识库，后续用户提问时可以检索到这条知识。

## 5. 最终效果

管理员可以在后台创建一个自动沉淀任务，例如：

```text
任务名称：每周客服知识沉淀
执行周期：每周一 02:00
数据范围：客服组最近 7 天的聊天记录
目标知识库：公司客服知识库
审核策略：人工审核后入库
提炼模型：gpt-4o-mini
最小置信度：0.75
```

任务到期后，系统自动读取符合条件的聊天记录，过滤掉空内容、闲聊、失败响应、测试消息和低价值片段，对候选对话进行脱敏与去重，然后调用 LLM 提炼为结构化知识。

生成结果进入“候选知识”列表。管理员可以看到问题、标准答案、分类、标签、来源聊天、置信度、重复风险和生成时间。管理员点击通过后，系统将内容写入目标知识库，并触发已有知识库处理流程，使其可以被后续 RAG 检索使用。

如果管理员拒绝某条候选知识，系统记录拒绝原因。后续任务遇到相似内容时，可以降低重复生成概率。

## 6. 核心业务流程

### 6.1 任务配置

管理员创建自动沉淀任务，配置任务名称、启用状态、执行周期、数据窗口、来源范围、目标知识库、提炼模型、审核策略和质量阈值。

来源范围至少支持按用户组筛选，因为真实公司场景里通常只希望沉淀客服组、运营组、销售支持组的业务对话，而不是全站所有用户聊天。后续可以扩展按用户、模型、频道、标签、文件夹筛选。

### 6.2 聊天采集

任务执行时，系统从 `chat_message` 表读取指定时间窗口内的消息。采集重点是完整问答上下文，而不是孤立消息。系统需要根据 `chat_id`、`parent_id`、`role` 把用户问题和助手回答组合成候选片段。

采集阶段只读取符合权限和范围的数据，不直接写入知识库。所有候选片段都带上来源信息，包括 `chat_id`、用户 ID、消息 ID、时间范围和模型 ID。

### 6.3 数据清洗

清洗阶段移除明显无价值内容，例如空消息、纯寒暄、格式错误、模型报错、未完成回答、重复生成内容、过短问答和测试会话。

系统还要做基础脱敏，至少识别并替换手机号、邮箱、身份证号、订单号、地址、API key 等敏感信息。MVP 阶段可以先做规则脱敏，后续再加入 LLM 辅助脱敏。

### 6.4 知识提炼

通过清洗的候选片段会交给 LLM 提炼。提炼目标不是总结聊天，而是生成可复用的企业知识条目。标准结构为：

```json
{
  "question": "客户如何申请退款？",
  "answer": "如果订单在 7 天内且服务未使用，客户可以在订单详情页提交退款申请...",
  "category": "售后/退款",
  "tags": ["退款", "售后", "订单"],
  "source_chat_ids": ["..."],
  "source_message_ids": ["..."],
  "confidence": 0.86,
  "risk_level": "low"
}
```

LLM 输出必须经过结构校验。缺少问题、答案、来源或置信度的结果不能进入审核队列。

### 6.5 去重与合并

系统要避免把同一个问题反复写入知识库。MVP 阶段可以使用标题/问题文本相似度、来源重复判断和向量相似度进行初步去重。

如果新候选知识与已有候选或正式知识高度相似，系统不直接新增，而是标记为“可能重复”。管理员可以选择忽略、替换、合并或作为新版本保存。

### 6.6 审核入库

默认策略是人工审核后入库。管理员在候选知识页面执行通过、拒绝、编辑后通过、合并、查看来源等操作。

审核通过后，系统将知识条目写入目标知识库。为了复用 Open WebUI 现有知识库处理链路，MVP 阶段建议将一批通过的知识条目生成 Markdown 文件，作为知识库文件写入，并走已有文件处理和向量化流程。

后续版本可以增加更细粒度的“知识条目”存储形态，但 MVP 不强行重构现有 Knowledge 模块。

## 7. 功能范围

MVP 必须包含以下能力：

- 管理员创建、编辑、启用、停用自动沉淀任务
- 支持每日和每周执行周期
- 支持按用户组选择聊天来源
- 支持选择目标知识库
- 支持从 `chat_message` 表按时间窗口采集问答
- 支持基础数据清洗和规则脱敏
- 支持调用任务模型提炼结构化候选知识
- 支持候选知识列表、详情、来源查看
- 支持管理员审核通过、拒绝、编辑后通过
- 支持审核通过后写入目标知识库
- 支持任务运行记录，包括成功、失败、耗时、输入消息数、候选知识数、入库知识数

## 8. 非目标与边界

MVP 不做全自动无审核入库。即使未来支持自动入库，也必须保留配置项和审计记录。

MVP 不做跨系统数据采集，例如企业微信、飞书、CRM、工单系统。这些可以作为后续扩展，当前只处理 Open WebUI 内部聊天记录。

MVP 不修改现有聊天主流程，不影响用户正常聊天响应速度。自动沉淀任务只能作为后台异步任务运行。

MVP 不做复杂知识图谱，不做多级本体建模，不做自动事实冲突解决。重复、冲突和低置信度内容交给审核流程处理。

MVP 不把用户原始聊天全文暴露给普通管理员以外的人。候选知识页面必须遵守管理员权限，来源查看也应限定为有权限的管理角色。

MVP 不承诺所有聊天都能沉淀为知识。系统应该允许任务产出为空，并把“没有高价值知识”视为正常结果。

## 9. 权限与安全要求

只有管理员可以创建和管理自动沉淀任务。

只有管理员或具备指定权限的知识库维护人员可以审核候选知识。权限命名建议为 `workspace.auto_knowledge` 或 `knowledge.auto_ingest`，具体命名在技术方案阶段与现有权限体系对齐。

任务配置必须明确数据来源范围，默认不采集全站聊天。推荐默认只能选择某个用户组，例如“客服组”或“运营组”。

所有候选知识必须保存来源引用，但正式入库内容默认不包含原始用户身份信息。敏感字段必须在入库前脱敏。

任务运行日志中不记录完整原始聊天内容，只记录必要统计和错误摘要。需要排查时再通过来源 ID 回查，并受权限控制。

## 10. 数据质量要求

候选知识必须满足四个基本条件：问题清晰、答案可复用、来源可追溯、内容已脱敏。

低置信度内容进入候选队列时要显著标识，不应默认通过。疑似重复内容也要标识，避免知识库膨胀。

系统要保留拒绝原因。拒绝原因既服务于人工管理，也可以为后续优化提炼 prompt 和去重规则提供依据。

知识入库后应能追溯到生成任务、审核人、审核时间和来源聊天。这样面试时可以清楚说明系统不是“黑盒自动总结”，而是有治理闭环。

## 11. 模块划分

后端建议分为五个部分。

第一部分是数据模型，负责自动沉淀任务、任务运行记录、候选知识、候选知识来源、审核记录。

第二部分是服务层，负责采集、清洗、脱敏、提炼、去重、入库等业务流程。

第三部分是调度执行层，负责按日/周触发任务。优先复用 Open WebUI 已有 scheduler worker 的模式，但不要把本功能硬塞进用户聊天 automations 语义里。Auto Knowledge 应该有自己的任务类型和运行记录。

第四部分是 API 路由，提供管理端配置、候选列表、审核操作、运行记录查询。

第五部分是前端管理页，提供任务配置、候选审核和运行日志。

各文件职责如下：

```text

  定义自动沉淀任务、运行记录、候选知识、来源映射和审核状态。

  暴露管理端 API，包括任务配置、候选知识审核和运行记录查询。

utils/auto_knowledge/scheduler.py
  接入后台调度循环，发现到期任务并执行。

utils/auto_knowledge/collector.py
  从 chat_message 表按时间、用户组、模型等条件采集候选问答。
```

## 13. 数据模型草案

MVP 建议新增四类表。

`auto_knowledge_job` 保存任务配置，包括任务名称、启用状态、执行周期、数据窗口、来源用户组、目标知识库、提炼模型、审核策略、质量阈值、上次运行时间和下次运行时间。

`auto_knowledge_run` 保存每次执行记录，包括任务 ID、状态、开始时间、结束时间、采集消息数、生成候选数、通过数、失败原因。

`auto_knowledge_candidate` 保存候选知识，包括问题、答案、分类、标签、置信度、风险等级、审核状态、目标知识库、生成任务、审核人和审核时间。

`auto_knowledge_source` 保存候选知识与来源聊天消息的映射，包括 candidate_id、chat_id、message_id、user_id、role、created_at。

## 14. 状态流转

候选知识状态建议如下：

```text
generated
  ↓
pending_review
  ↓          ↓
approved   rejected
  ↓
published
```

如果发布知识库失败，状态进入 `publish_failed`，允许管理员重试。

任务运行状态建议如下：

```text
queued
running
success
partial_success
failed
cancelled
```

`partial_success` 用于任务成功生成候选，但部分候选提炼失败或发布失败的情况。

## 15. 验收标准

管理员可以创建一个每周执行的知识沉淀任务，并指定用户组和目标知识库。

系统可以手动触发该任务，用于演示和测试。

任务执行后，可以从最近一段聊天记录中生成至少一条候选知识。

候选知识中必须包含问题、答案、分类、标签、置信度和来源消息。

管理员可以查看来源、编辑候选内容并审核通过。

审核通过后，候选知识进入目标知识库，并能被后续知识库检索命中。

如果聊天中包含手机号、邮箱、订单号等敏感信息，入库内容中必须被脱敏。

如果任务失败，运行记录页面能看到失败状态和错误摘要。

## 16. 项目经历表达方向

这个功能在简历里可以表达为：

```text
基于 Open WebUI 设计并实现企业运营知识自动沉淀模块，通过周期性后台任务从客服/运营聊天记录中抽取高价值问答，经过清洗脱敏、LLM 结构化提炼、去重合并和人工审核后写入 RAG 知识库，实现业务知识从日常对话到企业知识资产的自动闭环。
```

面试时重点讲四件事：为什么不能原始聊天直接入库，如何保证来源可追溯，如何用审核流程控制知识质量，为什么把它做成 Open WebUI 内置模块而不是插件或独立服务。

## 17. 后续扩展

后续可以支持自动入库策略，但必须限定低风险、高置信度、低重复度内容。

后续可以接入企业微信、飞书、CRM、工单系统，把外部运营数据也纳入沉淀范围。

后续可以增加知识版本管理，支持同一问题随政策变化产生新版本。

后续可以加入命中反馈：如果某条自动沉淀知识在后续问答中被频繁引用且被用户点赞，则提高可信度；如果被点踩或人工纠错，则进入复审队列。

后续可以增加冲突检测，例如新候选知识与已有知识答案不一致时，自动标记为“疑似冲突”，要求管理员处理。
